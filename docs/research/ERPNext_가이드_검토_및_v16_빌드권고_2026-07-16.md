## 2. 최우선 업데이트 — ERPNext v16 (문서 내용을 바꾸는 5가지)

### 2.1 버전 현황 (2026-07-16 기준)

| 항목 | 내용 |
|---|---|
| v16 정식 릴리스 | **2026-01-12** (당초 2025-12-06 → 연기) |
| 최신 패치 | **v16.28.0** (2026-07-15) — 6개월간 28회 패치, 초기 안정화 완료 단계 |
| v15 상태 | v15.117.0 — 유지보수 계속 중 |
| 지원 정책 | 동시 2개 메이저 버전 지원 (~2.5–3년 주기) → **v17 출시 시(예상 2027년경) v15 지원 종료** |
| v16 핵심 | Frappe Caffeine 아키텍처(~2x 성능), Espresso 신규 Desk UI, 보안 강화 |

### 2.2 문서 권고를 직접 바꾸는 v16 신기능

| # | v16 기능 | 문서의 어떤 내용이 바뀌나 |
|---|---|---|
| 1 | **Serial & Batch Traceability Report** (전방/후방 추적 내장 리포트) | 문서 ③⑦의 최대 커스텀 항목이었던 "원자재 로트→완제품→고객 추적 리포트"가 **표준 탑재**. 커스텀 범위는 '추적 리포트 개발' → 'HKMC 양식 맞춤 출력 + 리콜 시뮬레이션 절차'로 축소 |
| 2 | **전용 MRP 워크플로우** (수요예측·납품일정·리드타임 통합) | 문서 ② "Production Plan 내 제한적 MRP" 서술이 구버전 기준. OEM 포캐스트 기반 자재계획이 표준으로 가능 |
| 3 | **Stock Reservation for Work Orders** | 작업지시별 원자재 예약 → 양산 전환 시 결품·혼용 방지. 문서 ②③에 추가할 것 |
| 4 | **Financial Report Templates** (수식 기반 BS/P&L 커스텀) | 문서 ① "한국 재무제표 양식 커스텀" 부담이 노코드 수준으로 감소 |
| 5 | **Standard Cost 평가법 추가** (v16.28, 2026-07) | 문서 ③의 "FIFO/MA/LIFO/Standard 지원" 서술 — Standard Cost는 v15엔 없었고 **이번 달에야 추가됨**. 결과적으로 문서가 맞게 됐으나 v16.28+ 전제 |

**그 외 유관 신기능:** Inward Subcontracting(고객 사급자재 수취 가공 — HKMC 사급 구조에 해당 시 중요), 외주입고 Landed Cost Voucher, Automatic Closing Stock Posting, 월간 자동 재전기 옵션(Auto Repost Incorrect Valuation Entries), PDF 은행명세 임포트, 역할 기반 필드 마스킹(HR 급여정보 보호).

### 2.3 시스템 요구사항 변화 (빌드에 직결)

| 구성요소 | v15 | **v16 (필수)** |

| DB 옵션 | MariaDB만 | **SQLite 단독 실행 가능** (개발/실험용 — 운영은 MariaDB 유지) |

## 3. 섹션별 검토 (문서 ①~⑨)

### ① 회계 — ✅ 적절, 2건 보강
- COA 트리/차원 설계·"계정은 넓게, 세부는 차원으로" 원칙 모두 타당.
- **보강 1:** v16 Financial Report Templates로 K-GAAP 양식 재무제표를 수식 템플릿으로 구성 가능 → "커스텀" 표현을 "템플릿 구성"으로 완화.
- **보강 2 (핵심 의사결정 누락):** ERP가 **재무회계(법인세·부가세 신고)까지 대체**할지, **관리회계+운영(재고·생산·품질)만 담당**하고 세무는 기존 체계(더존/세무사)를 유지할지 — 이 범위 결정이 1단계 최우선인데 문서에 없음. 55명 규모 + 망분리 환경에선 **후자(운영 중심 ERP + 세무 이원화)가 현실적 초기안** (§5.1 참조).

### ② 제조 — ✅ 적절, v16 반영 필요
- BOM→WO→Job Card→backflush 흐름, 제출 BOM 불변 원칙 모두 정확.
- v16 전용 MRP·Stock Reservation·Inward Subcontracting 반영 (§2.2).
- **보강:** Job Card의 공정 실측 데이터(작업시간·불량수)가 PPAP 초기공정능력(Ppk) 증빙의 원천 — `reliability-alt-toolkit`의 통계엔진(Weibull 등)과 역할 분담 정의 필요: **ERP=데이터 수집·이력, toolkit=통계 분석·판정** (§6.6).

### ③ 재고/이력추적 — ✅ 적절, 1건 치명적 누락
- FIFO 권고, Allow Negative Stock=OFF, 창고 트리 사전 확정 — 모두 타당.
- **치명적 누락: Serial and Batch Bundle** (§4.3). v15부터 거래 문서의 serial/batch가 별도 Bundle DocType으로 분리 저장됨. 이를 모르고 커스텀 코드에서 `doc.serial_no`를 직접 읽으면 빈 값. **`babipa_erp`의 모든 추적 로직에 영향.**
- v16 Traceability Report 내장 반영 (§2.2). "리콜 시뮬레이션 정기 실시" 권고는 유지 — 리포트가 있어도 **절차·기록·소요시간 측정**은 조직이 해야 IATF 대응이 됨.

### ④ 구매 — ✅ 적절, EDI 현실화 필요
- 3-way match, Supplier Scorecard→IATF 공급사관리 연결 정확.
- CoC/MTR을 입고·로트에 링크하는 커스텀 — 타당. 구현은 **Purchase Receipt Item에 첨부+Batch 링크 커스텀 필드** 정도로 가볍게 시작 가능.
- **HKMC EDI = 실질적으로 VAATZ 포털.** 폐쇄망 ERP가 직접 연동 불가 → §5.1의 브리지 구조 필수.

### ⑤ 판매/CRM — ✅ 적절
- "OEM 단일고객 구조에선 CRM 비중 낮음" 판단 정확. **초기 스코프에서 CRM 모듈 제외(수주 SO부터 시작)**를 명시적으로 권고함이 나음 — scope creep 방지.

### ⑥ HR — ✅ 적절, 2건 보정
- 하이브리드 급여(국산SW 유지 + 회계 결과만 전기) 권고 타당. Frappe HR에 한국 4대보험·연말정산 로컬라이제이션은 여전히 없음.
- **보정 1:** HR은 v14부터 **별도 앱(`hrms`)** — ERPNext 설치만으로는 없음. apps.json에 명시적 추가 필요 (문서엔 이 구분이 없음).
- **보정 2:** v16 HR의 역할 기반 급여필드 마스킹·복수 휴일목록 배정은 55명 규모에서 바로 유용.

### ⑦ 품질 — ✅ 핵심 정확, 커스텀 범위 재조정
- IATF 8.5.2/PPAP 18요소/보존 10–15년 프레이밍 모두 타당.
- 커스텀 DocType 목록(PPAP·APQP·CP·PFMEA·FAI·4M)은 유지하되, **추적 리포트 부분은 v16 표준으로 대체** → 커스텀 총량 감소 = 문서 스스로 경고한 '과도한 커스터마이징' 리스크 완화.
- **보강:** Quality Inspection엔 **AQL 샘플링 플랜(KS Q ISO 2859 등) 자동화가 없음** — 수입검사 샘플 수 산정은 커스텀 스크립트 또는 검사기준서 수기 참조로 시작.

### ⑧ 기술 아키텍처 — ⚠️ 라이선스 서술 수정 필요
- hooks/override/Developer Mode/Virtual DocType 서술 정확.
- **수정:** "Frappe/ERPNext는 GPLv3(일부 AGPL)" →
  - **Frappe Framework = MIT** / **ERPNext = GPL-3.0** / **Frappe HR = GPL-3.0** / frappe_docker = MIT
  - AGPL-3.0은 **에코시스템 앱**(Insights·Builder·CRM·Drive 등)에 해당 — 이들을 설치하지 않으면 무관.
  - 실무 결론은 문서와 동일: **사내 인트라넷 전용 = 배포(conveying) 없음 = 소스공개 의무 미발생.** 화이트라벨 시 "ERPNext/Frappe" **상표·로고 사용 정책**만 별도 확인 (기능적 제거는 hooks로 가능).
- **보강:** v16 신규 Desk UI(Espresso)로 **구버전 URL 체계가 변경**됨 — v15 기준으로 작성된 커스텀 JS·북마크·매뉴얼 스크린샷은 전면 재검토 대상.

### ⑨ 도입 방법론 — ✅ 그대로 유효
- CSF 표·마스터데이터 최우선 원칙 모두 학술 근거와 일치. 수정 불요.

## 4. 옵션별 추가 고려사항

### 4.1 버전 선택: v15 vs **v16 (권고)**

| 기준 | v15 | **v16** |
|---|---|---|
| 지원 잔여기간 | v17 출시 시 종료 (~2027 추정) | **~2028+ (다음 두 사이클)** |
| 이력추적 | 커스텀 리포트 필요 | **내장 Traceability Report** |
| 마이그레이션 부담 | 나중에 v16 업그레이드 필수 (Python 3.14 재구축 포함) | 없음 (그린필드) |
| 안정성 | 검증 완료 | 릴리스 6개월+28패치 — **초기 리스크 구간 통과** |
| 리스크 | — | 커뮤니티 앱 중 v16 미지원 잔존 가능 → 서드파티 앱 도입 시 개별 확인 |

**판정: 신규 구축이므로 v16.** v15로 시작하면 1–2년 내 Python 3.14 + 신규 UI 이중 마이그레이션을 스스로 예약하는 셈. 단, **마이너 버전은 정확히 고정**(예: v16.28.x)하고 폐쇄망 특성상 검증된 태그만 반입.

### 4.2 평가방법: FIFO vs Moving Average vs Standard Cost

| | FIFO (권고) | Moving Average | Standard Cost (v16.28+) |
|---|---|---|---|
| 로트추적 정합 | ✅ 배치별 실원가 | ❌ 혼합단가 | △ 고정단가 |
| 소급전기 부담 | ⚠️ reposting 발생 | 낮음 | 낮음 |
| 추가 고려 | 소급 입력을 운영 규율로 최소화. v16.28의 **주간 자동 재전기 옵션** 활성화 검토 | — | **Purchase Price Variance 계정 필수** + 단가 변경 시 보유재고 재평가 전기 발생 — 표준원가 체계가 이미 있는 경우만 |

**판정: 문서 권고(FIFO) 유지.** 단 "소급 전표 입력 금지에 준하는 마감 규율"을 운영 SOP에 명문화할 것 — FIFO 최대 리스크는 기능이 아니라 소급 입력 습관.

### 4.3 Batch vs Serial — ⚠️ Serial and Batch Bundle 필수 이해

- 배분 원칙(안전부품=Serial, 범용=Batch)은 문서대로. 전장품은 **완제품 = Serial(DMC/바코드 단위), 원자재 = Batch(로트)** 조합이 일반적.
- **v15+ 구조 변경:** 거래 문서(Stock Entry, Delivery Note 등)의 serial/batch는 **`Serial and Batch Bundle`** 별도 DocType에 저장. 커스텀 코드 규칙:
  - ❌ `doc.items[0].serial_no` 직접 접근 (구버전 패턴 — 빈 값)
  - ✅ `serial_and_batch_bundle` 링크를 따라 Bundle의 entries에서 조회
- **양산 볼륨 경고:** 전수 Serial 부여 시 연 수십만 행의 Serial No 테이블 증가 → **어떤 품목을 Serial로 할지 'IATF 추적성 계획(traceability plan)' 문서에서 위험도 기반으로 먼저 결정**하고 ERP에 반영 (심사 시 이 문서 자체를 요구함).

### 4.4 급여: 하이브리드(권고 유지) vs 커스텀
- 문서 권고 유지. 추가 고려: 하이브리드 시 **월 1회 급여 분개 수동/반자동 전기 SOP** + 급여 데이터는 ERP 밖 → v16 필드 마스킹 이슈도 자연 회피. 커스텀 전환은 "ERP 정착 + 노무사 검증 리소스 확보" 이후로.

### 4.5 회계 범위: 재무회계 전체 vs 운영 중심 (신규 옵션)
- **옵션 A — 풀 재무회계:** 부가세·전자세금계산서까지 ERP. 폐쇄망에서 국세청/ASP 연동 불가 → 브리지 필수, 세무사 워크플로우 변경. 초기 부담 큼.
- **옵션 B — 운영 중심 (권고 초기안):** ERP = 재고·생산·품질·구매·판매 + 관리회계. 세무신고는 기존 체계 유지, 월마감 시 요약 대사.
- **판정: B로 Go-live → 안정화 후 A 확장 여부 재평가.** 문서 1단계의 "한국 COA·부가세 템플릿 설계"는 B에서도 필요하나 '신고 목적'이 아닌 '내부 정합' 수준으로 톤 다운.

### 4.6 PPAP/APQP: 커스텀 DocType(권고) vs 상용 QMS 병행
- 55명·단일 사업장·이미 Claude Code 툴킷 보유 → **커스텀 DocType이 합리적.** 상용 QMS(EwQIMS 등)는 라이선스비 + 폐쇄망 설치 제약 + ERP와 이중 데이터.
- 추가 고려: 커스텀 DocType 전부에 **Track Changes 활성화**(감사증적) + 문서 개정(Rev) 관리는 Frappe 표준 Amend 흐름 대신 **명시적 rev 필드 + 이전 rev 링크** 패턴 권장 (심사관이 개정이력을 한 화면에서 봐야 함).

## 5. 문서에 누락된 주제 (설계 전 반드시 결정)

### 5.1 망분리 ↔ 외부연동 모순의 해소 구조 ⭐최중요
문서는 "폐쇄망 배포"와 "HKMC EDI·전자세금계산서 연동"을 동시에 권고하지만 **직접 연동은 정의상 불가능.** 선택지:

| 구조 | 내용 | 판정 |
|---|---|---|
| **수동 브리지 (초기 권고)** | VAATZ 발주/납품 데이터를 외부망 PC에서 CSV/Excel 추출 → 승인 매체로 반입 → ERP **Data Import** 업로드. 역방향(납품 정보)도 동일 | 비용 0, 보안심사 무풍. 일 1–2회 배치로 충분한 규모 |
| DMZ 중계 서버 | 망간자료전송 솔루션/일방향 게이트웨이 경유 자동화 | 정확하나 55명 규모 대비 과투자. 물량 증가 시 재검토 |
| ERP 외부망 이중화 | 별도 외부 인스턴스 동기화 | ❌ 데이터 정합성 지옥 — 배제 |

→ `babipa_erp`에 **"VAATZ Import 매핑" 커스텀(CSV 컬럼→SO/DN 매핑 + 검증 스크립트)**을 3단계 커스텀 목록에 추가하는 것이 실질적 EDI 대응.

### 5.2 인쇄·라벨·한글 PDF (제조 현장 직결, 문서 전무)
- **PDF 엔진:** v16은 Print Settings에서 **wkhtmltopdf ↔ Chrome(headless)** 선택 가능. 한글 폰트·현대적 CSS는 Chrome 렌더러가 정답. 단 **공식 Docker 이미지에는 Chromium 미포함** → 커스텀 이미지 빌드 시 Chromium + **Noto Sans KR 폰트** 설치 레이어 추가 필요. 두 엔진은 **여백 모델이 달라** 양식이 상호 호환되지 않음 — 처음부터 Chrome 기준으로 통일.
- **필요 양식 목록화 (설계 단계):** 거래명세서, 시험성적서, 수입검사 성적서, 식별표(로트 라벨), 출하 라벨.
- **라벨 프린터:** Frappe **Raw Printing(ZPL/ESC-P)** 지원 — 현장 Zebra류 라벨기 직결 가능. 로트 라벨 = Batch QR/Code128 자동 인쇄가 이력추적 스캔 운영의 전제.

### 5.3 백업·DR + 10–15년 보존 전략
- `bench backup --with-files` 주기 실행(컨테이너엔 cron 없음 — **호스트 cron 또는 사이드카** 필요) + **NAS 등 물리적 분리 저장** + **분기 1회 복원 리허설**(백업은 복원 성공 전까지 백업이 아님).
- v16부터 원격백업이 **별도 Offsite Backup 앱**으로 분리 — 폐쇄망이라 불요하나, 마이그레이션 문서 읽을 때 혼동 방지용 인지.
- 10–15년 보존: SLE·Serial 테이블 증가 대비 **연 1회 아카이브 백업 영구보관 정책** + 디스크 증설 계획을 SOP에 명문화. (IATF 심사 시 "기록 보존 방법" 질문에 대한 답이 됨.)

### 5.4 권한·감사 설계 (Role Matrix)
- Go-live 전 **역할표(생산/품질/구매/경리/관리자) × DocType × 권한(RWCD+Submit/Cancel/Amend)** 매트릭스 확정. System Manager는 2인 이내.
- 폐쇄망이어도: 비밀번호 정책, 세션 만료, **운영 사이트 Server Script 비활성**(로직은 전부 앱 코드로 — RestrictedPython 제약 때문에도 앱 코드가 우월), 감사 대상 DocType Track Changes ON.

### 5.5 전자결재 갭
- Frappe Workflow = 상태 전이 + 승인자 지정 수준. 한국식 기안·합의·전결·대결은 미지원. **초기: 구매요청·발주 승인 등 핵심 2–3개만 Workflow로** 구성하고, 그룹웨어 수준 전자결재를 ERP에 요구하지 않기로 이해관계자와 합의(scope creep 1순위 후보).

### 5.6 폐쇄망 사이트 접근 설정 (사소하지만 첫날 막히는 지점)
- Frappe는 **Host 헤더 = 사이트명**으로 사이트를 식별. 인트라넷에서 IP로 접근하면 404 → frontend 컨테이너에 `FRAPPE_SITE_NAME_HEADER=<사이트명>` 고정 설정으로 해결. 사내 DNS(예: `erp.babipa.local`) 등록이 이상적.

### 5.7 테스트·CI (폐쇄망 제약)
- 외부 CI(GitHub Actions) 불가 → **이미지 빌드 전 개발 Mac에서 테스트 게이트 통과를 의무화**: `bench --site test_site run-tests --app babipa_erp` + pre-commit(ruff·prettier). Claude Code가 커밋 전 자동 실행하도록 CLAUDE.md에 명시 (§6.5).

## 6. Claude Code 빌드 가이드 (2026-07-16 기준)

### 6.1 환경 3단 구성

| 환경 | 위치 | 구성 | 용도 |
|---|---|---|---|
| **DEV** | MacBook | frappe_docker `development/` devcontainer (bench 대화형, `bench get-app` 가능) | Claude Code 개발 본진 |
| **STAGING** | 외부망 빌드 PC 또는 Mac | 운영과 동일한 커스텀 이미지 + compose | 반입 전 최종 검증·마이그레이션 리허설 |
| **PROD** | 사내 폐쇄망 서버 | 커스텀 이미지 (불변, `bench get-app` 불가) + compose | 운영 |

> Mac 네이티브 bench는 Python 3.14/MariaDB brew 조합 이슈가 잦음 — **처음부터 Docker devcontainer로 통일**하면 운영 이미지와 환경 편차도 사라짐.

### 6.2 서버 사이징 (55명 기준)
- 4 vCPU / **16GB RAM** / SSD 200GB+ (10년 보존 감안 증설 여지). MariaDB `innodb_buffer_pool_size`는 RAM의 40–50%.

### 6.3 apps.json + 커스텀 이미지 (버전 정밀 고정)

```json
[
  { "url": "https://github.com/frappe/erpnext", "branch": "version-16" },
  { "url": "https://github.com/frappe/hrms",    "branch": "version-16" },
  { "url": "https://{{ PAT }}@github.com/<org>/babipa_erp.git", "branch": "main" }
]
```
- `hrms`는 별도 앱 — 반드시 명시 (§3-⑥).
- 재현성이 중요하면 branch 대신 **정확한 태그**(`v16.28.0` 등) 고정. 폐쇄망에선 "검증한 그 이미지"가 전부다.
- `babipa_erp`는 private repo + PAT. (로컬 경로 COPY 방식도 가능하나 repo 방식이 이력 관리에 유리.)

```bash
# 외부망 빌드 PC (frappe_docker 클론 후)
export APPS_JSON_BASE64=$(base64 -i apps.json | tr -d '\n')   # macOS
docker build \
  --build-arg=FRAPPE_PATH=https://github.com/frappe/frappe \
  --build-arg=FRAPPE_BRANCH=version-16 \
  --build-arg=APPS_JSON_BASE64=$APPS_JSON_BASE64 \
  --tag=babipa/erpnext:16.28.0-b1 \
  --file=images/layered/Containerfile .
```

**커스텀 이미지에 추가 레이어 (한글 인쇄 대응, §5.2):**
```dockerfile
USER root
RUN apt-get update && apt-get install -y --no-install-recommends \
    chromium fonts-noto-cjk && rm -rf /var/lib/apt/lists/*
USER frappe
```
(빌드 후 Print Settings에서 PDF Generator=Chrome, chromium 경로를 site_config에 지정.)

### 6.4 폐쇄망 반입·설치·업데이트 파이프라인

```bash
# ① 외부망: 이미지 번들 생성 (compose가 참조하는 정확한 태그로 동반 반입)
docker save babipa/erpnext:16.28.0-b1 mariadb:10.6 redis:7-alpine \
  | gzip > erp_bundle_$(date +%Y%m%d).tar.gz

# ② 승인 매체로 반입 → 폐쇄망 서버
docker load < erp_bundle_YYYYMMDD.tar.gz

# ③ 최초 설치 (frappe_docker compose.yaml + overrides 반입본 사용)
docker compose up -d
docker compose exec backend bench new-site erp.babipa.local \
  --install-app erpnext --install-app hrms --install-app babipa_erp

# ④ 업데이트 절차 (매 반입 시 동일)
docker compose exec backend bench --site all backup --with-files   # 롤백 지점
# compose 이미지 태그 교체 → down/up
docker compose exec backend bench --site all migrate
```
- 업데이트 실패 시 롤백 = **이전 이미지 태그 + 직전 백업 복원.** 이 절차 자체를 STAGING에서 매번 리허설.
- frontend 서비스에 `FRAPPE_SITE_NAME_HEADER: erp.babipa.local` (§5.6).

### 6.5 CLAUDE.md에 넣을 규칙 (v16 대응 핵심)

```markdown
## 불변 원칙
- frappe/erpnext/hrms 코어 수정 절대 금지. 확장은 babipa_erp의 hooks.py
  (doc_events, override_doctype_class, fixtures)로만.
- 스키마 변경은 Developer Mode에서 → JSON으로 커밋. UI-only 변경 금지.
- Custom Field / Property Setter는 fixtures로 export (bench export-fixtures).
- 데이터 마이그레이션은 patches.txt + patch 모듈로.

## v16 Breaking Changes (코드 작성 시 준수)
- frappe.get_all / get_list는 Query Builder 백엔드 — v15식 필터 관용구 검증 필수.
- 기본 정렬이 modified → creation으로 변경 (리스트뷰·DB API 공통).
  정렬 의존 로직은 order_by 명시.
- 상태 변경 whitelisted method는 POST 전용 — JS에서 GET 호출 금지.
- sanitize_html이 nh3 기반 — 금지 태그는 통째로 제거됨 (이스케이프 아님).
- Report/Page JS는 IIFE 로딩 — 전역 변수 공유 금지.

## 데이터 접근 규칙
- Serial/Batch는 문서의 serial_and_batch_bundle 링크를 통해 조회.
  items[].serial_no 직접 접근 금지 (v15+ Bundle 구조).
- Server Script 사용 금지(운영 비활성) — 모든 서버 로직은 앱 코드.
  (RestrictedPython import 차단 — 기존 스킬 문서 유지)

## 품질 게이트 (커밋 전 자동 실행)
- bench --site test_site run-tests --app babipa_erp
- pre-commit run --all-files (ruff, prettier)
```

### 6.6 기존 `frappe-erpnext-toolkit` 업데이트 항목
1. **스킬 추가:** `v16-breaking-changes` (위 §6.5 내용 + 마이그레이션 위키 요지), `serial-batch-bundle` 조회 패턴.
2. **HKMC 컴플라이언스 에이전트:** 추적 요구 대응 로직을 "커스텀 리포트 생성" → "**표준 Traceability Report 활용 + HKMC 제출 양식 변환**"으로 갱신.
3. **`reliability-alt-toolkit` 연동 설계 확정:** 같은 인트라넷이므로 ERP→FastAPI REST 호출 가능. 역할 분담 — ERP(Job Card/QI)=원시데이터·이력, toolkit=Weibull/Arrhenius 분석. 분석 결과는 PPAP 관련 커스텀 DocType에 결과 JSON+PDF 첨부로 회수. (실시간 조회가 필요해지면 Virtual DocType 검토 — 초기엔 불요.)
4. **VAATZ Import 서브에이전트:** CSV 매핑→SO/DN 생성 + 검증 리포트 (§5.1).
5. UI 관련 슬래시 커맨드/스킬에 **v16 Espresso UI·URL 변경** 반영 (v15 스크린샷·경로 기반 문서 폐기).
